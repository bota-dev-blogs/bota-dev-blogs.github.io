## Project Overview

This project started with a practical goal: take Chinese-language GitHub tutorial videos and produce English versions that keep the original video track, technical meaning, timing, and speaker identity. The output should feel like the same tutorial in another language, not like a generic dubbed video pasted on top.

That requirement turns localization into a systems problem. Translation quality matters, but it is only one part of the stack. The pipeline also has to preserve segment timing, speaker boundaries, voice references, audio loudness, and final video synchronization.

![Project research map](/media/gifs/from-chinese-to-english-and-a-detour-into-moroccan-arabic/pipeline-3/01-article-summary.gif)

After the Chinese-to-English path worked, the project took a harder detour: could the same voice-cloning stack be pushed toward Darija, Moroccan Arabic? That experiment did not produce intelligible Darija speech, but it clarified where low-resource TTS fails: not in one dramatic bug, but in the interaction between data, orthography, speaker variation, and pronunciation ambiguity.

## The Main Pipeline

The working pipeline combines several components:

1. **Transcription** - [faster-whisper](https://github.com/SYSTRAN/faster-whisper) converts the Chinese audio into timed text segments.
2. **Translation** - [DeepL](https://www.deepl.com/) produces the first English translation, followed by a [Gemini](https://deepmind.google/technologies/gemini/) review pass for GitHub-specific terminology such as "pull request", "commit", and "branch".
3. **Speaker diarization** - [pyannote.audio](https://github.com/pyannote/pyannote-audio) identifies who spoke when, so each speaker can receive a separate reference voice.
4. **Speech synthesis** - [F5-TTS](https://github.com/SWivid/F5-TTS) and [Voxtral](https://mistral.ai/news/voxtral) generate English speech from translated segments and speaker references.
5. **Audio alignment** - [pyrubberband](https://github.com/bmcfee/pyrubberband) time-stretches long segments while preserving pitch; [pyloudnorm](https://github.com/csteinmetz1/pyloudnorm) normalizes loudness.
6. **Video merge** - [FFmpeg](https://ffmpeg.org/) replaces the original audio track with the translated, aligned, voice-cloned output.

![AI video language transformation pipeline diagram](/media/gifs/from-chinese-to-english-and-a-detour-into-moroccan-arabic/pipeline-2/01-ai-video-localization-pipeline.gif)

A [Streamlit](https://streamlit.io/) interface ties the workflow together. It lets the operator monitor each stage, inspect segment-level output, edit translations, and rerun weak parts of the pipeline.

## Why This Is More Than Translation

Text translation can ignore time. Video localization cannot.

If the English sentence is too long, it can overflow the original segment. If diarization assigns a segment to the wrong speaker, the voice clone breaks identity. If the reference clip contains noise, the synthetic voice inherits that weakness. If loudness varies between segments, the final video feels stitched together.

The usable unit is therefore not a sentence. It is a timed, speaker-labeled, quality-checked segment.

![Project explainer GIF](/media/gifs/from-chinese-to-english-and-a-detour-into-moroccan-arabic/pipeline-1/01-project-overview.gif)

That segment-level view also makes the system debuggable. When a clip sounds wrong, the operator can ask which layer failed: transcription, terminology, speaker assignment, voice reference quality, synthesis, timing, loudness, or merge.

## Voice Cloning Is The Hardest Link

Commercial TTS systems usually offer predefined voices. They are convenient, but they erase the original speaker. Zero-shot voice cloning is different: a model such as F5-TTS can synthesize new speech in a target speaker's voice using a short reference clip.

That makes the result feel more faithful, but it also raises the engineering bar. The system needs clean per-speaker reference audio, stable diarization, careful text segmentation, and enough timing flexibility to fit translated speech back into the source video.

The Chinese-to-English pipeline worked because those constraints were manageable. The source language was supported by ASR. English synthesis was supported by the TTS models. Technical terminology could be corrected by a review pass. Most importantly, the system could preserve the video timeline without asking the TTS model to learn a new language.

## The Darija Detour

Darija changed the problem.

Moroccan Arabic is the language of everyday life in Morocco: family conversations, street markets, jokes, voice notes, and informal online speech. It is also a difficult target for speech synthesis. It differs from Modern Standard Arabic in phonology, vocabulary, rhythm, and borrowing patterns from Amazigh, French, Spanish, and other sources.

It also has no single stable written form. Darija can be written in Arabic script, Latin script, or Arabizi, a Latin-numeric writing style where **3afak** can represent **عفاك** because "3" stands for the Arabic letter ع. Arabic-script Darija is usually unvocalized: short vowels are omitted, as in standard Arabic writing. That is normal for humans, but it creates ambiguity for a TTS model trying to learn pronunciation.

![Darija bottleneck map](/media/gifs/from-chinese-to-english-and-a-detour-into-moroccan-arabic/pipeline-3/02-darija-bottlenecks.gif)

## What We Tried

The fine-tuning attempt kept the F5-TTS acoustic architecture intact and changed the text input layer.

F5-TTS's base vocabulary contained 2,545 character tokens, but Arabic characters were not part of that vocabulary. Before training could begin, the vocabulary had to be extended.

The process was:

1. Collect every unique character from the Darija dataset.
2. Append new Arabic characters to the base vocabulary file.
3. Resize the model's text embedding matrix.
4. Initialize the new rows with random noise scaled to the standard deviation of existing embeddings.
5. Fine-tune on 20,000 samples from [DarijaTTS-clean](https://huggingface.co/datasets/KandirResearch/DarijaTTS-clean) on an H100 GPU in Google Colab.

```python
old_emb = state_dict["ema_model.transformer.text_embed.text_embed.weight"]
new_emb = torch.zeros(new_vocab_size, emb_dim)
new_emb[:old_size] = old_emb
std = old_emb.std().item()
new_emb[old_size:] = torch.randn(new_vocab_size - old_size, emb_dim) * std
```

The model produced speech-like rhythm and structure, but not intelligible Darija. It was generating audio, but it had not learned a reliable mapping from written input to Moroccan Arabic phonemes.

## Why The Attempt Failed

Two bottlenecks stood out.

**The data was too small and too spread out.** Teaching a voice model an underrepresented language is harder than adapting a voice inside a known language. The available Darija data was far below the likely data requirement for robust convergence, and it was distributed across multiple speakers. That forced the model to learn language structure and speaker variation at the same time.

**The writing system under-specified pronunciation.** Unvocalized Arabic-script Darija does not tell the model enough about vowels. Arabizi introduces a different normalization problem. Humans resolve those ambiguities through context and native knowledge. A TTS model needs either much more data or an explicit grapheme-to-phoneme layer.

## Quality Gates For Video Localization

The main pipeline points toward a useful architecture pattern for video localization: every stage should produce an inspectable artifact and a quality signal.

![Localization quality gates](/media/gifs/from-chinese-to-english-and-a-detour-into-moroccan-arabic/pipeline-3/03-quality-gates.gif)

| Stage | Artifact | Quality question |
| --- | --- | --- |
| Transcription | Timed source segments | Are words and boundaries correct enough to translate? |
| Translation | Target-language script | Does it preserve technical meaning and fit the segment? |
| Diarization | Speaker turns | Is each segment assigned to the right voice? |
| Voice cloning | Synthetic segment audio | Does it sound like the speaker without artifacts? |
| Alignment | Time-fitted audio | Does the speech fit without sounding unnaturally stretched? |
| Merge | Final video | Does the output feel like one coherent video? |

This is what made the Chinese-to-English path usable. The system did not depend on one perfect model. It depended on a chain of models whose outputs could be inspected, corrected, and recombined.

## What Comes Next

For Darija, the next step is probably not "train longer" by itself. A stronger path would combine:

- More hours of consistent, high-quality Darija speech.
- A smaller number of speakers for early convergence.
- Script normalization for Arabic script and Arabizi.
- A grapheme-to-phoneme layer that resolves missing vowels and loanword pronunciations.
- Evaluation clips organized by speaker, script, and text style.

The payoff would be significant. A good Darija TTS stack would not only support dubbing. It would also make voice interfaces, accessibility tools, and local-language educational content more realistic for Moroccan speakers.

## Closing Thought

The Chinese-to-English video pipeline worked: the visual track stayed intact, the translated audio was aligned, and the cloned voices made the output feel closer to the source.

The Darija experiment did not work, but it failed in a useful way. It showed that the missing piece was not only model architecture. It was data design, writing-system normalization, and pronunciation modeling.

Negative results are still engineering results when they narrow the next experiment.
